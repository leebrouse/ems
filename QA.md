论文答辩 QA（救援物资管理系统 EMS）

Q1：你的项目要解决什么问题？
A1：面向应急救援场景的物资管理平台，覆盖物资入库/出库、需求申请与调度、运输追踪与统计分析，解决跨角色协作中信息割裂、流程不透明与响应不及时的问题。

Q2：系统的总体架构是什么？
A2：采用前后端分离架构。前端为 Vue3 管理后台，后端采用 Go 微服务拆分（认证、用户、仓储、调度、统计），服务间通过 gRPC 调用，客户端通过 REST 接口访问。

Q3：为什么选择微服务拆分？
A3：不同业务模块职责清晰且耦合度低，微服务便于独立扩展与部署；同时配合 gRPC 实现服务间高效通信，降低单体耦合带来的维护成本。

Q4：前端技术栈与核心模块有哪些？
A4：Vue 3 + TypeScript + Vite，UI 使用 Element Plus，状态管理用 Pinia，路由用 vue-router；核心模块包括登录、概览统计、仓储管理、调度管理、用户管理和地图追踪。

Q5：前端权限控制如何实现？
A5：基于 RBAC。路由通过 meta.roles 控制可访问角色，菜单项根据用户角色动态过滤，未登录用户访问受限页面会被重定向到登录页。

Q6：后端有哪些服务？职责如何划分？
A6：auth 负责认证与 JWT；user 负责用户与角色；warehouse 负责物资与库存；scheduling 负责需求单、运输单与轨迹；statistics 聚合统计数据。每个服务提供 REST API，并对外/对内按需提供 gRPC。

Q7：认证与鉴权流程是什么？
A7：用户登录通过 auth 服务获取 JWT，前端将 token 持久化并注入到请求头；后端根据角色权限限制不同接口的访问范围。

Q8：调度与运输的业务流程如何串联？
A8：需求单由调度模块创建与指派，生成运输任务；运输任务在地图上展示路线和状态；相关统计由统计服务聚合，形成面向管理者的决策看板。

Q9：数据存储与 ORM 方案是什么？
A9：使用 PostgreSQL 作为主数据库，ORM 采用 GORM，模型按服务划分，保证数据域边界清晰。

Q10：为什么选择 REST + gRPC 双通道？
A10：REST 便于前端消费与联调，gRPC 用于服务间高效通信，提升内部调用性能并保证接口契约一致。

Q11：系统可观测性如何设计？
A11：接入 OpenTelemetry，统一 OTLP 上报，Collector 汇聚后分别对接 Jaeger 和 Prometheus，支持链路追踪与指标监控。

Q12：部署方案是什么？
A12：Docker Compose 一键拉起数据库与后端各服务，前端本地启动并通过代理访问后端。也支持独立构建各服务镜像。

Q13：系统的主要创新点是什么？
A13：将应急救援场景的需求单、运输任务、库存预警与统计可视化做成闭环流程；前后端在同一权限模型下联动，结合地图轨迹提升调度透明度。

Q14：项目中遇到的关键技术难点？
A14：多服务之间的数据一致性与状态流转控制，以及多角色权限在前后端的统一约束。通过明确领域边界、统一接口与 RBAC 规则解决。

Q15：数据一致性如何保证？
A15：业务上尽量遵循单一服务对核心数据的唯一写入原则，跨服务通过 RPC 获取或聚合；对于状态流转，采用明确的状态枚举与校验规则。

Q16：前端如何处理后端字段差异？
A16：在前端对响应进行规范化处理，兼容字段大小写或命名差异，确保页面渲染稳定。

Q17：安全性如何考虑？
A17：认证使用 JWT，接口按角色限制访问；敏感配置通过配置文件集中管理，避免硬编码在代码中。

Q18：性能与可扩展性如何保障？
A18：服务拆分可横向扩展，gRPC 提升内部通信效率；前端请求统一封装，避免重复逻辑；统计服务独立出来减少核心业务压力。

Q19：测试与验证如何开展？
A19：前端采用 TypeScript 类型检查，后端可通过接口测试与服务联调验证，部署后通过日志与链路追踪观察运行状态。

Q20：未来工作计划有哪些？
A20：补齐自动化测试、完善权限管理的细粒度策略、支持多租户或多组织管理、引入更精细的库存预测与调度优化模型。
